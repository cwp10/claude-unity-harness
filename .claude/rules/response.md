# 응답 규칙

## 소통
- 한국어로 소통
- 코드 주석은 한국어 + 영어 병기

## 주석 작성 원칙 (공통) — 기본값은 "주석 없음"
- **기본 원칙: 주석을 쓰지 않는다.** 쓰고 싶을 때 먼저 "정말 필요한가?"를 자문할 것
- 다음 경우에만 예외적으로 허용:
  - 숨은 제약 / 비자명한 불변조건 / 특정 버그 우회 / 놀라운 동작
  - public API의 `/// <summary>` XML 문서 주석
- **절대 금지 (있으면 즉시 제거):**
  - 코드가 이미 말해주는 **What** 설명 (`// i를 1 증가`, `// 플레이어 이동`)
  - 잘 명명된 식별자의 의미 재설명
  - 현재 작업·수정·호출자 언급 (`// X 작업으로 추가`, `// Y 플로우에서 사용`, `// issue #123 대응`)
  - 진행 상황·TODO가 아닌 결정 기록 (`// 이렇게 바꿈`, `// 기존 방식 제거`)
  - 섹션 구분용 장식 주석 (`// ===== 초기화 =====`)
  - 주석 처리된 (죽은) 코드 — 즉시 삭제, git이 기억함
  - 파라미터·리턴값을 그대로 옮긴 docstring
- **판정 기준:** 이 주석을 지웠을 때 미래 독자가 혼란스러울 것인가? 아니라면 쓰지 말 것
- 길이: 꼭 써야 한다면 한 줄. 여러 줄 주석 블록·다문단 docstring 금지
- 기존 코드 수정 시, 발견한 불필요한 주석은 같이 정리할 것

## 구현 전 원칙
- 규모가 Medium 이상이면 설계 플랜 먼저 제시 후 승인 대기
- 리스크가 있으면 반드시 먼저 언급
- 불확실하면 가정하지 말고 질문
- 대안이 있으면 2가지 이상 제시

## 다이어그램
- 모든 흐름도·다이어그램은 Mermaid 형식 사용 (ASCII 아트 금지)
- **Mermaid 버전: 8.8.9** — 플러그인 호환 버전 고정
  - `graph TD/LR` 사용 (`flowchart` 키워드 사용 금지)
  - 사용 가능: `-->`, `---`, `-.->`, `==>`, `subgraph`, `classDef`

## 금지 사항
- 확인 없이 파일 삭제 금지
- 확인 없이 대규모 리팩토링 금지
- 코드 변경 후 테스트 방법 안내 생략 금지
- 성능에 영향을 주는 변경 시 반드시 명시
